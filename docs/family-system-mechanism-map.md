# Family System Mechanism Map

## Purpose

This map captures how `Family System v13.yaml` created strong behavior in Kilo. It extracts mechanisms, not bulk prompt text, so Manurella can inherit the useful operating pressure without becoming another giant runtime-specific YAML.

Source audited:

- `C:\Users\rehan\Desktop\Projects\Logic Equation Solver\Family System v13.yaml`

## Core Finding

Family System v13 worked because it behaved like an operating system, not because it had many named agents. Its strongest output came from the combination of:

- high-pressure identity
- explicit routing and state rules
- strict handoff schemas
- memory ownership
- quality gates
- failure recovery
- domain instinct libraries
- few-shot anti-pattern correction

Manurella must preserve those mechanisms in modular form.

## Load-Bearing Mechanisms

### 1. Definition Style

Family agents were defined with this structure:

- identity and authority
- conversational behavior
- origin story or operating reason
- aesthetic or professional conviction
- workflow protocols
- domain mastery layer
- stack/model/runtime-specific layer
- instincts, red flags, and anti-patterns
- output format or ticket schema

Manurella adaptation:

- Keep this as the domain-agent authoring pattern.
- Do not reduce agents to plain job titles.
- Put long domain knowledge in references, but keep behavioral pressure in agent definitions.

### 2. Orchestrator As Evaluator

Brioni was not only a router. She had cross-domain gut checks and could reject specialist output before routing forward.

Manurella adaptation:

- `manurella-orchestrator` must classify, route, evaluate, and stop bad handoffs.
- Core must own universal quality review.
- Domain leads must expose "excellent / red flag / gut check" standards.

### 3. Task Classifier

Family System separated quick tasks, feature additions, full builds, conversation, ambiguous requests, onboarding, resume, learning, and change-impact work.

Manurella adaptation:

- Add universal task classes to the core protocol.
- Fast Mode is not lower quality. It is a quick-task envelope.
- Standard Mode is for multi-step work with verification.

### 4. Handoff Ticket

The old system forced delegated work into a structured ticket with mission, focus, references, acceptance criteria, blockers, architecture mode, and fast-track status.

Manurella adaptation:

- Runtime packets and handoff packets must have required fields.
- Delegation without a bounded packet is invalid.
- Exact errors and evidence must not be paraphrased away.

### 5. Memory Ownership

The memory bank had owners, readers, required fields, and append/overwrite rules. That prevented random memory writes and helped continuity.

Manurella adaptation:

- Use `cognition/graph.yaml`, eval records, learner state, and future project-state files as owned memory surfaces.
- Every durable memory change needs owner, evidence, and scope.
- No automatic immortal memory without review.

### 6. Project State Machine

Family System routed differently for greenfield, salvage, audit, reimagine, sprint, resume, and onboarding.

Manurella adaptation:

- Core must classify project state before Build work.
- Existing projects must not receive greenfield routing.
- Audit mode must not write code unless explicitly converted to a fix task.

### 7. Output Quality Review Gate

Before accepting specialist output, Brioni ran gut checks first, then checklists. If the gut check failed, the output failed.

Manurella adaptation:

- Every routed task needs a quality gate.
- The orchestrator may reject a specialist result once and request a focused repair.
- After repeated failure, escalate with the gap instead of looping.

### 8. Loop And Blocker Control

Family System treated repeated mutual blockers and unresolved relays as routing failures requiring human decisions.

Manurella adaptation:

- Detect repeated repair loops, circular delegation, and unresolved blockers.
- Do not send the same weak packet again.
- Escalate with exact evidence and options.

### 9. Context Hygiene

Family System had explicit compression, file-size, and ticket-size rules.

Manurella adaptation:

- Keep always-on context small.
- Use packet references and graph retrieval instead of embedding giant manuals.
- Summarize long runs into durable records before continuing.

### 10. Domain Instinct Libraries

The old agents contained immediate standards: what excellent output looks like, what shallow output looks like, and the one gut-check question that exposes weakness.

Manurella adaptation:

- Each domain lead needs an instinct library.
- Build gets engineering gut checks.
- Muse gets craft and continuity gut checks.
- Pixel gets visual intent and output-fidelity gut checks.
- Mentor gets evidence-of-learning gut checks.

## Deprecated Patterns

Do not carry these forward directly:

- one huge runtime YAML as source of truth
- runtime-specific Kilo assumptions in the kernel
- large always-on knowledge dumps
- agent identities that depend on decorative names instead of contracts
- silent model switching
- unbounded parallel agents
- writing durable memory without validation

## Baseline Implication

The true Manurella v0 baseline is not complete until:

1. The root brain contains Family-level operating pressure.
2. The orchestrator has task classification, handoff, quality review, recovery, and state handling.
3. Domain leads follow the Family definition pattern.
4. Kilo export preserves these behaviors.
5. At least one old-vs-new eval shows Manurella matching or exceeding Family System behavior.
